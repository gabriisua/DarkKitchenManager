<objective>
Consolidate all 401/403 error handling into error.interceptor.ts and strip it from auth.interceptor.ts.

**Current problem:** Two interceptors handle 401/403. Because errorInterceptor runs after authInterceptor in the response chain (Angular interceptor order: api → auth → error → HTTP, so errors flow error → auth), the error interceptor's unconditional `handleAuthFailure()` fires first for ALL 401/403 — including login failures. The auth interceptor's `isPublic` guard is effectively dead code. Public-endpoint requests like login *should* be allowed to fail without triggering a logout/redirect.

**Solution:** Remove 401/403 logic from auth.interceptor.ts (it should only attach tokens). Move the public-endpoint guard to error.interceptor.ts so it runs first and correctly skips login/reset-password/forgot-password failures.

**Output:** Two clean interceptors — one does token attachment only, the other does error handling with public-endpoint awareness.
</objective>

    import { inject } from '@angular/core';
    import { catchError, throwError } from 'rxjs';
    import { AuthService } from '../services/auth.service';

    export const errorInterceptor: HttpInterceptorFn = (req, next) => {
      const authService = inject(AuthService);
      const publicEndpoints = ['login', 'reset-password', 'forgot-password'];

      return next(req).pipe(
        catchError((error: HttpErrorResponse) => {
          if (error.status === 401 || error.status === 403) {
            const isPublic = publicEndpoints.some(url => req.url.includes(url));
            if (!isPublic) {
              authService.handleAuthFailure();
            }
          }
          return throwError(() => error);
        })
      );
    };
